# 银行零售业务智能营销决策支持系统前端设计

## 1. 目标与定位

本前端不是单纯展示“分类预测结果”，而是服务于课程作业中的“决策支持系统”表达。

首页目标有两个：

1. 让老师一眼看出这是一个面向银行零售营销的 DSS，而不是单模型预测页面。
2. 让演示过程可以顺着“筛选客户 -> 查看候选名单 -> 查看单客户决策建议 -> 查看 LLM 解释”自然展开。

本次前端先聚焦首页工作台，不在这一轮设计中展开登录、权限、后台配置等非核心页面。

## 2. 设计结论

首页采用 `标准工作台` 方案：

- 顶部：摘要指标带
- 左侧：筛选器面板
- 中间：客户候选名单
- 右侧：单客户决策详情

其中最重要的设计约束是：

`LLM 解释`、`推荐动作`、`建议话术` 必须严格绑定到“当前选中的某一个客户”，不能绑定在客户群体画像上。

## 3. 页面信息架构

### 3.1 顶部摘要指标带

用途：

- 服务答辩与汇报表达
- 快速展示系统整体价值
- 帮助用户理解当前客户池规模与营销资源配置结果

建议指标：

- 待评估客户数
- 高潜客户数
- 预估转化率
- 建议营销预算

设计原则：

- 顶部摘要只做“全局总览”
- 不承载单客户解释
- 视觉上比工作区更轻，避免抢占主叙事

### 3.2 左侧筛选器面板

用途：

- 用于定义当前决策范围
- 支持用户按业务条件缩小客户池

建议筛选项：

- 客户价值层级
- 渠道偏好
- 年龄区间
- 接触月份
- 职业类别
- 风险状态或贷款状态

交互原则：

- 允许多条件筛选
- 筛选条件变化后，驱动中间客户名单刷新
- 左侧只做条件输入，不展示复杂解释

### 3.3 中间客户候选名单

用途：

- 展示当前筛选条件下的候选客户池
- 承担“选择一个客户进入决策详情”的入口作用

每条客户记录建议展示：

- 客户编号或匿名标识
- 模型预测概率
- 预期收益等级或优先级
- 推荐渠道

交互原则：

- 点击某一客户后，右侧详情区整体刷新
- 中间列表是“候选对象池”，不是最终结论页
- 默认高亮第一位或综合评分最高的客户

### 3.4 右侧单客户决策详情

这是首页最核心的区域。

建议拆成 5 个块：

1. 客户基本信息
2. 模型输出结果
3. 推荐动作
4. LLM 解释
5. 建议话术

具体说明如下。

#### 3.4.1 客户基本信息

展示当前选中客户的关键特征，例如：

- 年龄
- 婚姻状态
- 职业
- 房贷/个人贷状态
- 最近联系月份

要求：

- 只展示与决策相关的关键字段
- 避免把整个原始记录堆满页面

#### 3.4.2 模型输出结果

展示内容建议：

- 转化概率
- 综合等级判断，例如“高响应倾向”
- 可选展示预期收益等级

要求：

- 这一部分先呈现定量判断
- 为后面的推荐动作和 LLM 解释提供依据

#### 3.4.3 推荐动作

这是 DSS 的决策输出，不是模型概率的重复表达。

建议展示：

- 是否优先触达
- 推荐触达渠道
- 推荐产品方向
- 推荐触达时机

示例：

“建议在 48 小时内电话触达，优先推荐 6 个月定期存款产品。”

#### 3.4.4 LLM 解释

这部分必须解释“为什么给这个客户这个建议”，而不是泛泛解释某类客户。

解释输入建议来自：

- 当前客户特征
- 当前模型预测结果
- 当前业务规则结果

输出风格要求：

- 使用自然语言
- 简明、业务化、可直接用于汇报
- 聚焦该客户的可营销原因与风险

示例表达：

“该客户在相似人群中表现出较高的存款接受倾向，且联系间隔适中，没有明显过度打扰风险，因此本轮触达的边际收益较高。”

#### 3.4.5 建议话术

这部分与 `LLM 解释` 分开。

区别是：

- `LLM 解释` 是给系统使用者看的
- `建议话术` 是给营销执行者参考的

示例：

“您好，我们注意到您更适合稳健型资金配置，目前有一款期限合适、收益稳定的存款方案，可能比较符合您的资金安排需求。”

## 4. 页面叙事顺序

首页演示顺序固定为：

1. 看顶部摘要，说明系统整体价值
2. 在左侧切换筛选条件
3. 在中间看到候选客户名单变化
4. 点击一个客户
5. 在右侧查看该客户的模型输出、推荐动作、LLM 解释和建议话术

这样可以把系统讲成一个完整的 DSS 流程，而不是“模型打分页面”。

## 5. 视觉方向

视觉上采用“轻商务 + 暖色数据工作台”方向。

核心特征：

- 米白、浅砂、墨蓝、深青为主色
- 页面整体偏专业、克制，不做炫技式大屏
- 信息密度中等，适合课堂演示和报告截图
- 顶部摘要与右侧详情有明确层级差异

不采用的方向：

- 纯深色大屏风格
- 过强科技蓝紫风格
- 过度卡片化、像通用后台模板的风格

## 6. 组件划分建议

如果后续用 Streamlit 实现，首页建议拆成如下逻辑模块：

- `summary_metrics`
  - 渲染顶部摘要指标
- `filter_panel`
  - 管理筛选条件
- `customer_list`
  - 展示客户候选名单与选中状态
- `customer_detail`
  - 展示单客户信息、模型结果、决策建议
- `llm_explanation_card`
  - 渲染单客户解释
- `llm_script_card`
  - 渲染建议话术

拆分目的：

- 页面逻辑清晰
- 后续同学分工更方便
- 后面接模型与 LLM 时边界明确

## 7. 数据映射建议

基于 `bank-additional-full.csv`，前端首页第一版可直接使用这些字段：

- 年龄：`age`
- 职业：`job`
- 婚姻：`marital`
- 教育：`education`
- 是否违约：`default`
- 房贷：`housing`
- 个人贷：`loan`
- 联系方式：`contact`
- 最近联系月份：`month`
- 上次通话时长：`duration`
- 本次活动联系次数：`campaign`
- 历史联系间隔：`pdays`
- 历史联系次数：`previous`
- 历史结果：`poutcome`
- 目标变量：`y`

说明：

- `duration` 在严格建模时可能有信息泄漏争议，但在前端展示与课堂报告中可以作为分析字段讨论
- 前端第一版可以先用静态样例或预处理结果，不要求先完成全部模型接入

## 8. 演示场景建议

建议准备一个标准演示路径：

1. 默认进入高价值客户筛选
2. 系统展示若干候选客户
3. 点击一位高概率客户
4. 右侧展示该客户的推荐动作与 LLM 解释
5. 切换到另一位客户，展示解释内容如何随客户变化

这个步骤能最直接证明：

- 系统不是静态报告
- LLM 输出是“逐客户变化”的
- 决策支持逻辑是动态的

## 9. 非目标范围

这一轮前端设计暂不包含：

- 用户登录页
- 权限管理
- 多角色工作流
- 真实在线 API 部署
- 完整数据库管理界面
- 多页复杂报表系统

避免首页承担过多功能，保证课程作业能稳定落地。

## 10. 后续实现优先级

建议按下面顺序实现：

1. 先做静态首页布局
2. 再接入筛选与客户切换交互
3. 再接入模型输出结果
4. 最后接入 LLM 解释与建议话术

原因：

- 先把演示骨架跑通
- 再逐步替换成真实数据和真实模型结果
- 可以降低联调风险

## 11. 验收标准

前端首页设计完成后，应满足：

1. 能明显看出这是 DSS，而不是单一预测页面
2. 首页结构清楚呈现“筛选器 + 客户列表 + 单客户详情”
3. 右侧的 `LLM 解释` 与 `建议话术` 明确绑定当前客户
4. 页面适合课堂演示截图与口头汇报
5. 后续能够平滑接入 Streamlit、模型结果和 LLM 输出
